## When to use

Reach for this skill whenever you need to decide how a URL or web-rendered page should appear inside (or alongside) your Android app. The right answer turns on two axes: who owns the content, and how much native control you need. Common entry points include: opening a help article, embedding a checkout flow, shipping a progressive web app through the Play Store, or displaying a hybrid UI that mixes web and native elements.

## Core guidance

### The four options at a glance

| Approach | Content owner | Browser chrome | Access to native APIs | Maintenance surface |
|---|---|---|---|---|
| Custom Tabs | External (third-party) | User's browser | None (sandboxed) | Zero |
| Trusted Web Activity (TWA) | You (PWA you control) | None — full-screen | Delegated via Digital Asset Links | PWA codebase |
| WebView | You (controlled origin) | Custom or none | JS Bridge / WebMessageListener | High |
| Native re-implementation | N/A | N/A | Full | Highest |

### Custom Tabs — the safe default for external links

Use Custom Tabs any time you are opening a URL you do not own: OAuth flows, payment provider pages, help-centre articles, social-share previews, or any third-party site. Benefits:

- The page renders in the user's installed browser (Chrome, Firefox, Samsung Internet); cookies, saved passwords, and autofill carry over automatically.
- No `INTERNET` permission handling or security review burden for the remote content — the browser owns the sandbox.
- Seamless UX: the toolbar colour adapts to your app's brand via `setToolbarColor` / `setColorScheme`, and the back gesture returns the user to your app.
- Partial Custom Tab (`setInitialActivityHeightPx`) shows content as a bottom sheet, useful for previews or quick info pages without fully leaving the app.
- Warm-up and pre-fetching (`CustomTabsClient.warmup`, `mayLaunchUrl`) eliminate cold-start latency when the URL is predictable.

Do not use Custom Tabs for content your app owns and needs to control (intercept requests, inject JS, read cookies programmatically). That is WebView territory.

### Trusted Web Activity — shipping a PWA as an app

TWA is the right path when your team has already built (or is willing to maintain) a PWA that meets Play Store quality standards, and you want it distributed as a first-class app. Key points:

- The entire viewport is your PWA; there is no browser toolbar, address bar, or back button from the browser — it looks fully native.
- Verified via Digital Asset Links: the `/.well-known/assetlinks.json` file on your domain must match the app's signing certificate. Without this, the browser toolbar reappears, breaking the experience.
- The host APK is thin — often generated by Bubblewrap or PWABuilder — and the real surface is the web codebase.
- Suitable when: web-first team, single codebase across platforms, content-heavy apps where web rendering is sufficient.
- Not suitable when: you need deep native features (camera pipeline, Bluetooth LE, sensors), offline-heavy scenarios with complex local state, or high-frame-rate rendering (games, AR).

### WebView — owned hybrid UIs

Reach for `WebView` (embedded in a `Composable` via `AndroidView`) when you own and control the web content and need either: (a) a hybrid UI that mixes web-rendered content with native chrome, (b) JavaScript bridging to call Kotlin APIs from web code, or (c) a fully offline web app bundled in `assets/`.

Key practices:

- Enable `WebViewAssetLoader` to serve local assets over a real `https://` scheme, avoiding the mixed-content and `file://` security pitfalls.
- Use `WebMessageListener` (API 23+) or `addJavascriptInterface` for the JS bridge; annotate all bridge methods with `@JavascriptInterface` and restrict origins explicitly.
- Disable `setAllowFileAccessFromFileURLs` and `setAllowUniversalAccessFromFileURLs` (both off by default — do not re-enable).
- Keep WebView up to date via the Android System WebView module; do not hard-code an assumed Chromium version.
- For Compose integration, wrap `WebView` in `AndroidView` and manage state (URL, back-stack) in a ViewModel; handle the Android back gesture by intercepting `BackHandler` and calling `webView.goBack()` when `canGoBack()` is true.

```kotlin
@Composable
fun EmbeddedWebView(url: String, modifier: Modifier = Modifier) {
    var webViewRef by remember { mutableStateOf<android.webkit.WebView?>(null) }
    val canGoBack = remember { mutableStateOf(false) }

    BackHandler(enabled = canGoBack.value) {
        webViewRef?.goBack()
    }

    AndroidView(
        factory = { ctx ->
            android.webkit.WebView(ctx).apply {
                settings.javaScriptEnabled = true   // only if your content requires it
                webViewClient = object : android.webkit.WebViewClient() {
                    override fun onPageFinished(view: android.webkit.WebView, url: String) {
                        canGoBack.value = view.canGoBack()
                    }
                }
            }.also { webViewRef = it }
        },
        update = { it.loadUrl(url) },
        modifier = modifier
    )
}
```

### Native re-implementation — when web is the wrong layer

Sometimes the answer is to not use any web technology. Build natively when:

- The UI requires platform-level performance: 60/120 fps animations, custom gesture recognisers, complex Canvas/RenderNode drawing.
- You need full integration with Android APIs: Bluetooth LE, NFC, camera2/CameraX pipeline, media codecs, accessibility services.
- The web page you were going to embed is simple enough to replicate with a `Column`, `LazyColumn`, and a few `Text` and `Image` composables — the overhead of a WebView or Tab is not justified.
- Content is entirely local and does not require a web rendering pipeline.

Native re-implementation is the most expensive path but the one that gives you the strongest UX fidelity, the best accessibility tree, and the most predictable performance.

### Decision summary

```
Does the URL belong to a third party?
  → Yes  → Custom Tabs (warm up early for speed)
  → No
     Do you maintain a production-quality PWA already?
       → Yes  → TWA (set up Digital Asset Links, ship thin APK)
       → No
          Do you need JS bridging, request interception,
          or bundled offline HTML?
            → Yes  → WebView (own the content, lock down settings)
            → No
               Is the content simple enough to build in Compose?
                 → Yes  → Native re-implementation
                 → No   → WebView (bundled or same-origin)
```

## Platform notes

**Large screens and foldables** — Custom Tabs support `setInitialActivityHeightPx` and work correctly in multi-window. WebView inside `AndroidView` adapts to size changes if you pass a `Modifier` that responds to `fillMaxSize` or window-size classes; test resizing explicitly. TWA in desktop windowing mode (ChromeOS) works well because the browser engine already handles responsive layouts.

**Android 16 edge-to-edge** — from Android 15, edge-to-edge enforcement is mandatory. WebView content should include the appropriate viewport meta tag and CSS safe-area insets. Custom Tabs and TWA handle this automatically.

**WebView version** — on API 26+ the WebView is an updatable module from the Play Store. Do not assume a specific Chromium version; use feature detection in JS rather than version sniffing.

**Custom Tabs availability** — if no browser supporting Custom Tabs is installed (rare, but possible on locked-down enterprise devices), fall back to `Intent.ACTION_VIEW`. The `CustomTabsClient.getPackageName()` helper returns `null` in this case.

**TWA Digital Asset Links** — verification happens at install time and can be cached. Test with `adb shell pm get-app-links --user 0 <package>` to confirm the verified state before release.

## Pitfalls

- **Using WebView for external links** — opening third-party URLs in a WebView means your app inherits their security surface. Mixed-content errors, phishing pages, and ad-network scripts all run in your process. Use Custom Tabs instead.
- **Forgetting `WebViewClient`** — without a `WebViewClient`, redirects open in the device browser, splitting the session. Always set one if you need self-contained navigation.
- **JavaScript bridge injection scope** — `addJavascriptInterface` exposes the entire annotated object to every frame in the WebView. Restrict with `WebMessageListener` and explicit origin allowlists when possible.
- **TWA without verified links** — if `assetlinks.json` is misconfigured, the browser toolbar appears and the app looks broken. Automate verification checks in CI with the `digital-asset-links` API.
- **Custom Tabs session not warmed up** — calling `launchUrl` cold on a slow device can feel slower than `ACTION_VIEW`. Always call `CustomTabsClient.warmup` during app start if you anticipate opening a Custom Tab.
- **WebView back-stack leak in Compose** — not intercepting the system back gesture causes the OS to finish the Activity instead of navigating back within the WebView. Use `BackHandler` as shown above.
- **Targeting `file://` origins** — loading `file://` pages and enabling `setAllowFileAccessFromFileURLs` creates a path traversal vulnerability. Use `WebViewAssetLoader` with an `https://` scheme instead.
